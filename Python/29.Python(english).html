so we will learn basics of database design here to get you started. 
the Purpose of database design is to for it to be easily understook and make things work together. Whether you're managing customer information, product catalogs, or employee records, a well-structured database ensures that data is organized, easily accessible, and, most importantly, reliable.
, the process involves understanding the requirements of your application and translating them into a logical structure that can efficiently store and retrieve data. Think of it as building the foundation of a house – you want it to be solid and capable of supporting everything that rests upon it.

One of the fundamental tools used in the database design process is the Entity-Relationship (ER) diagram. This visual representation helps us understand the relationships between different entities/tabels (refers to a distinct/unique object or concept about which data is stored in the database. ) in our database and how they interact with each other.

Entities, in this context, represent the various objects or concepts we need to store information about – they could be anything from customers and orders to products and suppliers. Relationships, on the other hand, describe how these entities are connected to each other. For instance, a customer places an order, creating a relationship between the customer entity and the order entity.

Now, let's tie it all together with an example. Imagine you're designing a database for an online bookstore. You'd likely have entities such as books, authors, customers, and orders. The relationships between these entities would be crucial – for instance, a customer can place many orders, but each order belongs to only one customer.
By creating an ER diagram for our bookstore database, we can visualize these relationships and ensure that our database design accurately reflects the requirements of our application. This not only helps us in the initial stages of development but also serves as a valuable reference as the project evolves.
In essence, the ER diagram serves as our compass

# Creating the relationship:
# In SQL, you would use a foreign key to establish the relationship between these tables.
# For example, in the CPUs table, 'type_id' is a foreign key that references the 'id' in the CPU_Types table.

***SQL*** 

CREATE TABLE CPU_Types (
    id INT PRIMARY KEY, -- I willl tell about types of keys at the end 
    type VARCHAR(50)
);
INSERT INTO CPU_Types (id, type) VALUES (1, 'AMD'), (2, 'Intel');


CREATE TABLE CPUs (
    id INT PRIMARY KEY,
    model VARCHAR(50),
    type_id INT,
    FOREIGN KEY (type_id) REFERENCES CPU_Types(id)
);
INSERT INTO CPUs (id, model, type_id) VALUES 
(1, 'Ryzen 5 5600G', 1),
(2, 'Ryzen 7', 1),

(5, 'i9 14th gen', 2);

-- View all data in CPU_Types
SELECT * FROM CPU_Types;

SELECT * FROM CPUs;

--This query joins the CPUs table with the CPU_Types table to show the manufacturer name instead of the type_id.
SELECT CPUs.id, CPUs.model, CPU_Types.type 
FROM CPUs
JOIN CPU_Types ON CPUs.type_id = CPU_Types.id;

Let me tell you about Keys
Candidate Key :-> The minimal set of attributes that can uniquely identify a tuple is known as a candidate key. For Example, STUD_NO in STUDENT relation

Primary Key:-> There can be more than one candidate key in relation out of which one can be chosen as the primary key. For Example, STUD_NO, as well as STUD_PHONE, are candidate keys for relation STUDENT but STUD_NO can be chosen as the primary key (only one out of many candidate keys). 

Alternate Key :-> The candidate key other than the primary key is called an alternate key.

Foreign Key :-> If an attribute can only take the values which are present as values of some other attribute, it will be a foreign key to the attribute to which it refers. The relation which is being referenced is called referenced relation and the corresponding attribute is called referenced attribute. The referenced attribute of the referenced relation should be the primary key to it.
STUD_NO in STUDENT_COURSE is a 
foreign key to STUD_NO in STUDENT relation

Composite Key :-> Sometimes, a table might not have a single column/attribute that uniquely identifies all the records of a table. To uniquely identify rows of a table, a combination of two or more columns/attributes can be used.  It still can give duplicate values in rare cases. So, we need to find the optimal set of attributes that can uniquely identify rows in a table. the combination is called composite key
FULLNAME + DOB can be combined 
together to access the details of a student.

Logical Key :-> A key that the "real world" might use to look up a row. For example, in a data model, a user's screen name might be a logical key that the program uses to look up their row multiple times.
